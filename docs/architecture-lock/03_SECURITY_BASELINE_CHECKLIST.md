# Hisab Security Baseline Checklist

Version: 1.0 (Architecture Lock)
Date: 2026-04-10
Scope: Frontend app, backend API, and data stores

## 1. Security Decision Log (Locked)

## 1.1 Password Hashing Strategy (bcrypt vs argon2)

Decision matrix:

| Criterion | bcrypt | argon2id |
| --- | --- | --- |
| Node.js backend support maturity | High | High |
| Expo managed app compatibility without custom native pipeline | Medium-High (via JS libs) | Low-Medium (often requires native integration) |
| Memory-hard resistance | Medium | High |
| Immediate implementation complexity for 2-person team | Low | Medium-High |
| Current code migration risk | Low-Medium | Medium |

Final decision for v1.0:
1. Use `bcrypt` for backend and local credential hashing in v1.0 for implementation certainty with current stack.
2. Use per-user random salt generated by bcrypt.
3. Backend salt rounds: `12`.
4. Local device salt rounds: `10` (performance on low-end devices).
5. Re-evaluate migration to `argon2id` in v1.2 after moving to custom native build.

Mandatory controls:
1. Never store plaintext passwords.
2. Never log password input or hash values.
3. Enforce minimum password length 10 for production builds (6 allowed only in non-production test mode).

## 1.2 Token Policy

Access token:
1. Type: JWT (`token_type=access`)
2. TTL: 15 minutes
3. Claims required: `iss`, `aud`, `sub`, `jti`, `iat`, `exp`, `token_type`

Refresh token:
1. Type: JWT (`token_type=refresh`)
2. TTL: 7 days
3. Rotation: required on every refresh
4. Storage (server): hashed token only
5. Reuse detection: mandatory; revoke all active sessions on detected reuse

Revocation rules:
1. Password change revokes all active refresh tokens
2. Logout revokes submitted refresh token
3. Inactive refresh tokens cleaned by scheduled job

## 1.3 Secret Management

Runtime secret sources:
1. Development: `.env.local` (never committed)
2. Production: managed secret store (cloud secret manager or vault)

Mandatory secrets:
1. `JWT_ACCESS_SECRET`
2. `JWT_REFRESH_SECRET`
3. `JWT_ISSUER`
4. `JWT_AUDIENCE`
5. `MONGO_URI`
6. `ENCRYPTION_MASTER_KEY`

Key rotation policy:
1. Rotation interval: every 90 days
2. Emergency rotation: within 2 hours of compromise suspicion
3. JWT key version via `kid` claim/header
4. Maintain previous key for grace period equal to max token lifetime

## 1.4 API Protection Baseline

1. Rate limiting by route class:
   - Auth login: 10 requests / 15 min / IP
   - Auth refresh: 40 requests / 10 min / IP
   - Mutations: 120 requests / 10 min / user
   - Reads: 300 requests / 10 min / user
2. Request schema validation is mandatory at controller boundary.
3. Reject unknown fields on mutation endpoints.
4. Ownership enforcement mandatory in query filters and updates.
5. Security headers:
   - `X-Content-Type-Options: nosniff`
   - `X-Frame-Options: DENY`
   - `Referrer-Policy: no-referrer`
6. CORS allowlist by environment; wildcard disallowed in production.

## 1.5 Data Protection Baseline

At rest:
1. MongoDB disk encryption enabled in production environment.
2. Local SQLite sensitive columns encrypted using AES-256-GCM envelope encryption where feasible.
3. Tokens and encryption key material stored only in secure device keychain/keystore.

In transit:
1. TLS 1.2+ mandatory.
2. Reject plaintext HTTP in production.

PII policy:
1. Mask phone numbers in logs (`+88017****678`).
2. Audit logs should avoid raw secrets and password-related material.

## 3. Threat Scenarios and Required Mitigations

1. Stolen refresh token replay:
   - Mitigation: refresh token hashing + rotation + reuse detection + global revocation
2. Offline tampering with local DB:
   - Mitigation: signed sync operations, secure key storage, server validation on ingest
3. Cross-user data exposure by API bug:
   - Mitigation: repository-level ownership filters + automated isolation tests
4. Brute-force login attempts:
   - Mitigation: rate limiting + lockout + alerting
5. API payload injection:
   - Mitigation: strict schema validators and server-side sanitization

## 4. Security Acceptance Criteria (Release Gate)

All must pass before v1.0:
1. 0 critical and 0 high open security defects.
2. All SEC-01..SEC-15 controls implemented or explicitly risk-accepted by product owner.
3. Auth abuse scenarios produce expected lockout/rate-limit behavior in tests.
4. Multi-user isolation tests pass with zero leakage.
5. Secret rotation drill completed successfully once in staging.

## 5. Placement and Integration Requirements

Frontend placement:

Backend placement:
1. Centralize auth config and validation middleware.
2. Add structured security event logging (`security_events` collection).

Database placement:
1. Add indexes supporting ownership and token revocation lookups.
2. Add retention policies for old auth artifacts and expired sessions.
